# Roadmap: Apps Script backend + Lovable frontend

## Зачем этот roadmap

Наша цель - перестать править красивый интерфейс вручную в Apps Script и перейти к нормальной схеме:

```text
Apps Script = данные и расчёты
Lovable = красивый frontend
GitHub = единое хранилище проекта
```

## Этап 1. Зафиксировать backend

Статус: начато.

Что уже сделано:

- код Apps Script сохранён в GitHub;
- общий период `dateFrom/dateTo` добавлен в логику текущего дашборда;
- деньги в `gsdashboard.gs` считаются через `Сумма` + `Вид операции`;
- структура проекта лежит в `apps-script/`.

Что нельзя менять без отдельного решения:

- ID Google Sheets;
- названия листов;
- структуру колонок;
- бизнес-логику расчётов.

## Этап 2. Подготовить контракт для Lovable

Статус: текущий этап.

Файлы:

```text
docs/data-contract.md
docs/lovable-frontend-brief.md
docs/roadmap.md
```

Цель: Lovable должен понимать, какие данные уже есть и какой frontend надо строить.

## Этап 3. Сделать визуальный frontend на mock data

Первый шаг для Lovable:

- не подключаться сразу к Apps Script;
- не трогать Google Sheets;
- не менять расчёты;
- собрать красивый frontend на mock data, полностью повторяющий контракт.

Ожидаемый результат:

- современный mobile-first dashboard;
- 4 раздела: Дэшборд, Деньги, Прибыль, Капитал;
- общий период;
- понятные KPI;
- разные типы графиков;
- хороший первый экран для руководителя.

## Этап 4. Подключить frontend к backend

После утверждения дизайна:

- определить способ вызова Apps Script из внешнего frontend;
- при необходимости добавить JSON endpoint;
- сохранить контракт ответов;
- добавить обработку ошибок;
- проверить CORS / публикацию web app.

## Этап 5. Публикация

Возможные варианты:

- Lovable hosting;
- Vercel;
- GitHub Pages;
- отдельный поддомен.

Решение принимается после того, как frontend будет готов и понятно, как он обращается к Apps Script backend.

## Не делать сейчас

Не делать сейчас:

- не переносить вручную все файлы обратно в Apps Script без необходимости;
- не делать Lovable сразу поверх старого HTML;
- не смешивать frontend React и Apps Script HTML в одном файле;
- не менять таблицу данных;
- не начинать публикацию до утверждения визуального прототипа.

## Следующий конкретный шаг

1. Создать PR с документацией для Lovable.
2. Смержить документацию в `main`.
3. Дать Lovable промпт из `docs/lovable-frontend-brief.md`.
4. Попросить Lovable сначала сделать frontend-прототип на mock data.
