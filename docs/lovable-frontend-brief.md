# Бриф для Lovable: красивый frontend финансового дашборда

## Цель

Сделать современный, понятный и красивый финансовый дашборд для руководителя / CFO.

Backend и расчёты остаются в Google Apps Script. Lovable делает frontend: внешний вид, UX, адаптивность, структуру экранов, визуальную подачу KPI и графиков.

## Главная идея продукта

Пользователь заходит в дашборд и за 10 секунд понимает:

1. есть ли деньги;
2. прибыльна ли компания;
3. есть ли риск по капиталу и ликвидности;
4. что изменилось за выбранный период;
5. куда смотреть дальше.

## Текущие разделы

Frontend должен сохранить 4 основных раздела:

```text
Дэшборд
Деньги
Прибыль
Капитал
```

## Общий период

В интерфейсе должен быть общий период:

```text
С даты
По дату
```

Правила:

- период общий для всего приложения;
- при переходе между разделами период не сбрасывается;
- `Деньги`, `Прибыль`, `Дэшборд` используют `dateFrom/dateTo`;
- `Капитал` использует `dateTo` как дату среза;
- в `Капитале` изменение `dateTo` не должно сбрасывать `dateFrom`.

## Главный экран: Дэшборд

Задача главного экрана - не показать всё, а быстро ответить на вопрос: состояние компании хорошее или нет.

Рекомендуемая структура:

1. Верхняя строка:
   - общий период;
   - дата среза капитала;
   - предупреждение, если срез капитала найден раньше выбранной даты.

2. Первый блок KPI:
   - ЧДП за период;
   - Cash Buffer;
   - Чистая прибыль;
   - Рентабельность;
   - Ликвидность;
   - Собственный капитал.

3. Блок интерпретации:
   - зелёный статус: всё стабильно;
   - жёлтый статус: есть риск;
   - красный статус: критично.

4. Графики:
   - ЧДП по месяцам;
   - Выручка + MR%;
   - Чистая прибыль + рентабельность;
   - ДЗ / КЗ;
   - структура капитала.

## Экран Деньги

Главная задача: показать движение денег и риск кассового разрыва.

Нужные блоки:

- ЧДП за период;
- Cash Buffer;
- стартовый остаток;
- расход за период;
- ЧДП по месяцам;
- расходы по статьям;
- ЧДП по направлениям;
- контрагенты;
- таблица ОДДС.

UX-важно:

- не перегружать экран одинаковыми графиками;
- денежные потоки визуально отличать цветом;
- расходы не маскировать, показывать явно;
- на телефоне KPI должны читаться без горизонтального скролла.

## Экран Прибыль

Главная задача: показать экономику бизнеса.

Нужные блоки:

- выручка;
- валовая прибыль;
- MR%;
- чистая прибыль;
- рентабельность;
- прибыль по направлениям;
- динамика по месяцам;
- структура ОПиУ.

## Экран Капитал

Главная задача: показать устойчивость компании на дату среза.

Нужные блоки:

- собственный капитал;
- ликвидность;
- финансовая независимость;
- структура активов;
- структура пассивов;
- динамика дебиторской задолженности;
- динамика кредиторской задолженности;
- заемный vs собственный капитал.

## Визуальный стиль

Направление: современный CFO dashboard.

Требования:

- тёмная тема допустима, но контраст должен быть высоким;
- денежные показатели должны быть визуально важнее подписей;
- не делать все графики одинаковыми;
- использовать разные способы подачи: KPI-карты, линии, бары, stacked bars, donut, таблицы;
- минимум лишнего текста;
- первый экран должен быть понятен за 10 секунд;
- мобильная версия обязательна.

## Что нельзя ломать

Lovable не должен:

- менять структуру Google Sheets;
- менять бизнес-расчёты;
- придумывать новые API поля без согласования;
- использовать колонки `Приход` и `Расход` в деньгах;
- делать frontend, который работает только на desktop;
- заменять общий период отдельными периодами на каждой странице.

## Техническая целевая архитектура

```text
Google Sheets
   ↓
Google Apps Script backend
   ↓ JSON/API contract
Lovable / React frontend
   ↓
Публикация красивого дашборда
```

Apps Script пока может оставаться текущим web app. Следующий этап - сделать для frontend удобный JSON endpoint, если текущий `google.script.run` станет неудобен для внешнего сайта.

## Первый промпт для Lovable

```text
Создай современный mobile-first frontend финансового дашборда CFO на основе документа docs/data-contract.md и docs/lovable-frontend-brief.md.

Не меняй бизнес-логику и структуру данных. Сначала сделай визуальный прототип на mock data, полностью повторяющий контракт данных backend.

Разделы: Дэшборд, Деньги, Прибыль, Капитал.

Главное требование: на первом экране руководитель за 10 секунд должен понять состояние компании: деньги, прибыль, ликвидность, капитал, главный риск.

Сделай разные типы визуализации, не одинаковые графики везде. Обязательно mobile-first.
```
