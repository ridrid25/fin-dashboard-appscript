# Контракт данных финансового дашборда

## Назначение

Этот документ описывает, какие данные backend на Google Apps Script отдаёт будущему красивому frontend в Lovable / React.

Цель: не переписывать бизнес-логику в Lovable. Lovable должен делать интерфейс, визуализацию, UX и адаптивность. Apps Script остаётся источником данных и расчётов.

## Источник данных

Данные читаются из Google Sheets через `apps-script/gscore.gs`.

Текущий источник:

```text
DATA_SPREADSHEET_ID = 1jTjAlrBClUhAnhUqKsaVZ_CjZf7657oSeebGB8KVrto
```

Листы:

```text
Деньги
Капитал
Прибыль
```

Именованные диапазоны:

```text
rangecash
rangeequity
rangeprofit
```

## Общий формат ответа API

Все публичные функции должны возвращать объект одного типа:

```json
{
  "status": "success",
  "data": {
    "kpis": {},
    "charts": {},
    "tables": {},
    "meta": {}
  },
  "error": null
}
```

Ошибка:

```json
{
  "status": "error",
  "data": null,
  "error": "Текст ошибки"
}
```

## Общий период

Глобальный период хранится на frontend стороне:

```text
financeDashboard.dateFrom
financeDashboard.dateTo
```

Формат дат:

```text
YYYY-MM-DD
```

Правило:

- `dateFrom` и `dateTo` используются в разделах `Дэшборд`, `Деньги`, `Прибыль`;
- `Капитал` использует только `dateTo` как дату среза;
- при переходах между разделами период не должен сбрасываться.

## Деньги

Публичная функция:

```javascript
cashApiMoney(params)
```

Параметры:

```json
{
  "dateFrom": "2023-01-01",
  "dateTo": "2023-12-31",
  "account": null,
  "counterparty": null,
  "article": null,
  "direction": null
}
```

Колонки листа `Деньги`:

```text
Дата
Счет
Контрагент
Статья
Раздел ОДДС
Вид операции
Направление
Сумма
```

Критически важная логика:

- отдельные колонки `Приход` и `Расход` не используются;
- `Сумма` содержит значение операции;
- `Вид операции = Приход` означает положительный денежный поток;
- `Вид операции = Расход` означает отрицательный денежный поток.

## Прибыль

Публичная функция:

```javascript
profitApiProfit(params)
```

Базовые параметры:

```json
{
  "dateFrom": "2023-01-01",
  "dateTo": "2023-12-31"
}
```

Раздел используется для ОПиУ и расчёта:

- выручки;
- себестоимости;
- валовой прибыли;
- маржинальности;
- чистой прибыли;
- рентабельности.

## Капитал

Публичная функция:

```javascript
capitalApiCapital(params)
```

Параметры:

```json
{
  "dateTo": "2023-12-31"
}
```

Логика:

- ищется последний доступный срез капитала на дату `dateTo` или ранее;
- `dateFrom` не участвует в расчёте капитала;
- изменение даты в капитале не должно сбрасывать общий `dateFrom`.

## Дэшборд

Публичная функция:

```javascript
dashboardApiDashboard(params)
```

Параметры:

```json
{
  "dateFrom": "2023-01-01",
  "dateTo": "2023-12-31"
}
```

Логика:

- `dateTo` используется для поиска среза капитала;
- `dateFrom` используется как начало периода для денег и прибыли;
- если `dateFrom` не передан, fallback: 1 января года даты среза;
- деньги считаются по `Сумма` + `Вид операции`, без старых колонок `Приход`/`Расход`.

Meta дэшборда:

```json
{
  "dateFrom": "2023-01-01T00:00:00.000Z",
  "dateTo": "2023-12-31T00:00:00.000Z",
  "snapshotDate": "2023-12-31T00:00:00.000Z",
  "periodStart": "2023-01-01T00:00:00.000Z",
  "periodEnd": "2023-12-31T00:00:00.000Z",
  "warning": null
}
```

## Что Lovable не должен делать

Lovable не должен:

- менять Google Sheets;
- менять расчётную бизнес-логику;
- придумывать новые поля без согласования;
- читать таблицу напрямую;
- заменять Apps Script backend на свою логику;
- использовать старые колонки `Приход` и `Расход`.

## Что Lovable должен делать

Lovable должен:

- сделать красивый frontend;
- улучшить мобильный UX;
- показать KPI и графики понятнее;
- использовать готовый контракт данных;
- сохранять общий период;
- показывать ошибки API понятным языком;
- отделить визуальную часть от расчётной логики.
